# 1.3.0

> ### **Release Highlights**
> 
> - New Support for On-Premise Environments
> - Addition of Existing Tibero Integration Features and New Build Features
> - Support for RMGR-Based Backup/Recovery Framework

# New Features

## On-Premise Environment Support

OwlDB is now supported for use in On-Premise environments.
Database deployment and operational management are also available within customer internal infrastructure environments.

## Database Registration and Installation

### Database Registration

Existing Tibero instances in operation can be registered in OwlDB for management.
Registered databases can use the same operational features in the OwlDB console, including status monitoring, parameter management, and backup/recovery.

### Database Installation

An automated installation feature is supported to enable new Tibero deployments in On-Premise environments.
By entering the required configuration information during the installation process, everything from database creation to initial setup is performed collectively, allowing a standardized architecture-based operating environment to be configured quickly.

### Change Detection

The change detection feature allows you to identify database status changes and specification changes that occurred outside of OwlDB, and apply the updated configuration information back to OwlDB.

## Account and Permission Management

### Account Management

A feature to create and manage user accounts has been added.

Access and usage permissions for databases managed in OwlDB can be configured per account.

### Permission Management

Feature access can be controlled based on the Root and Member permission hierarchy.
Depending on the assigned permission, the ability to perform key operations such as database creation and deletion is restricted.

- Root: Administrator permission that allows performing database operations and key management tasks.
- Member: General user permission that allows viewing assigned databases and using limited operational features.

## Failback

A Failback feature has been added to restore the switched operating environment to its original configuration after a Failover.
After a failover event, the Primary and Standby roles are re-established to restore the service operating configuration to its normal state.

# **Changes**

## Feature Improvements

### Backup/Recovery

**RMGR-Based Backup Support**

RMGR-based backup is supported in On-Premise environments.

- Cloud environment: CSP Snapshot-based
- On-Premise environment: RMGR-based

By applying the appropriate backup method according to the environment, backup management suited to the database operating environment is possible.

**Backup Scheduler**

Full Backup and Incremental Backup can be configured separately in the backup scheduler.

- Execution intervals and retention periods can be set per backup type to configure a backup policy suited to the operating environment.
- Full Backup: Performs a backup based on the entire dataset.
- Incremental Backup: Performs a backup based on data changed since the previous backup.

**Backup Chain Management**

The relationship between Full Backup and Incremental Backup can be viewed in units of chains from the backup list.
Incremental Backups are linked and managed in association with their base Full Backup.

## Removed Features

### Backup Scheduler

**Removal of the 'Auto Archiving' Configuration Feature**

The auto archiving configuration feature has been removed to prevent unnecessary accumulation of backup files.
