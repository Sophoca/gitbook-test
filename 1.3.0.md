# 1.3.0

> ### **Release Highlights**
> 
> - New On-Premise Environment Support
> - Addition of Existing Tibero Integration Features and New Build Features
> - RMGR-Based Backup/Recovery System Support

# New Features

## On-Premise Environment Support

OwlDB is now newly supported for use in On-Premise environments.
Database deployment and operational management is now possible within customer internal infrastructure environments.

## Database Registration and Installation

### Database Registration

Existing Tibero instances in operation can be registered and managed in OwlDB.
Registered databases can use the same operational features in the OwlDB console, including status monitoring, parameter management, and backup/recovery.

### Database Installation

An automated installation feature is supported to enable new Tibero deployments in On-Premise environments.
By entering the required configuration information during the installation process, everything from database creation to initial setup is performed in a single operation, allowing a standardized architecture-based operational environment to be configured quickly.

### Change Detection

The change detection feature allows you to identify database status changes and spec changes that occurred outside of OwlDB, and reflect the updated configuration information back into OwlDB.

## Account and Permission Management

### Account Management

A feature for creating and managing user accounts has been added.

Database access and usage permissions managed in OwlDB can be configured on a per-account basis.

### Permission Management

Feature access can be controlled based on the Root and Member permission system.
Depending on the permission level, the ability to perform key operations such as database creation and deletion is restricted.

- Root: Administrator permission that allows performing database operations and key management tasks
- Member: General user permission that allows viewing assigned databases and using limited operational features

## Failback

A Failback feature has been added to restore the operational environment switched after a Failover back to its original configuration.
After a failover, the Primary and Standby roles are reassigned to restore the service operational configuration to its normal state.

# **Changes**

## Feature Improvements

### Backup/Recovery

**RMGR-Based Backup Support**

RMGR-based backup is supported in On-Premise environments.

- Cloud environment: CSP Snapshot-based
- On-Premise environment: RMGR-based

By applying the appropriate backup method for each environment, backup management suited to the database operational environment is possible.

**Backup Scheduler**

Full Backup and Incremental Backup can be configured separately in the backup scheduler.

- The execution interval and retention period can be set per backup type to configure a backup policy suited to the operational environment.
- Full Backup: Performs a backup based on the entire dataset.
- Incremental Backup: Performs a backup based on data changed since the previous backup.

**Backup Chain Management**

The relationship between Full Backup and Incremental Backup can be viewed as a chain unit in the backup list.
Incremental Backups are linked and managed together with the Full Backup they are based on.

## Removed Features

### Backup Scheduler

**Removal of the 'Auto Retention' configuration feature**

The auto retention configuration feature has been removed to prevent unnecessary accumulation of backup files.
